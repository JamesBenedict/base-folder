IDS Build-Out Template

This is a general template to publish long form articles at the Indiana Daily Student. If you have any questions, feel free to come talk to me at the IDS Mondays and Wednesday from 3 - 4 p.m. or email me at jabenedi@indiana.edu.

This template is adapted from Waiting for Help published in the Spring of 2015 at the Indiana Daily Student. It is being developed for stories from Bonnie Layton's multimedia section of the Words and Pictures course at IU in the Fall of 2015.

Read and follow all comments in the html if you wish to publish at the IDS. Call in ids.css and ids.js last in your page unless you are using another plugin that requires its file to be loaded last. If something isn't changing after you style it in the style.css, it might be that the ids.css is over riding it. If this is the case, you can ask to alter the ids.css style, but you need a justification.

This template was developed to provide a starting point for designers looking to publish their first responsive article. I tried to keep it simple and have it serve as a skeleton for your design. Try to come up with some ideas that differentiate your article from the rest, but keep the overall feel the same. This index.html has every currently available feature, so it will take a while to load and might lag. Make sure to start from the starter folder and do not edit this one, because if you do your article might have uneeded javascript and media files which could cause lag.

Getting Started

Download the template folder from github. You can either download the folder as a zip or setup github and fork it. If you're confused about github and want to use it come into office hours. Github is very powerful and has a lot of cool features that I have no idea really how to use. I just use it for backing up my code and making it easier to share. This level of understanding is pretty easy to figure out.

There will be a starter folder in the template. Pull it out of the template folder and rename it your story’s slug. Don't change the template folder’s files but keep it as a resource to copy paste from. If anything does break, you can always redownload the folder. The starter folder only has the basics in it to keep it light and load fast.

Try to only work on a few things at a time and work on them until they are functional. Don't leave a bunch of half finished features you plan on finalizing later. Since this is a collaborative project make sure to comment your code well on anything you don’t think makes sense.

I normally start by placing my section heads and linking them into the nav bar. Then I move onto placing the first draft of the copy, before starting to layout the media.

The story’s structure will change around during the editing process so don't get too caught up on the exact layout of everything. Instead try to break the story down into modules that are easy to copy and paste around. Sometimes this is base on the article's sectoin heads, sometimes it is based around graphics. You know your story best and can make the call. I often have a comment before and after code modules that might switch around so I quickly can figure out where it starts and where it ends. Think about building with legos not bricks.

Media Categories

There are two categories your media falls into. Figure media requires a figure tag around it and a figcaption to explain what it is. HTML media doesn't require a figure tag although it can be place in one if you want. HTML media is place in custom divs you or I have made. In general, html content scales size easier, but figure content is quicker to setup.

Figure media includes

  • images
  • videos
  • audio
  • embedded content

HTML media includes

  • body copy
  • pulled quotes
  • more info box
  • related content box
  • slideshows

  • stylized text

Media Placement

The grid of this article on a medium or larger screen is one body-copy column that is 7 columns wide with one column offset to the left, and one sidebar column that is 4 columns wide. For mobile, everything is the full 12 colunmns wide. These columns live in a row which can be used to organize sections of your code. You can copy in as many columns into a row as you want, but once the total width of the columns equals 12, any other columns will float underneath. Since everything floats to the top, the easiest way to lineup the sidebar with the body-copy is to start a body-copy column on the paragraph that you want to line up the sidebar with. Then include the sidebar underneath and the sidebar will float and line up with the top of the first paragraph or first media in the body-copy column. If you want to read more about the grid structure, read about Bootstrap scaffolding.

Any media can be placed in one of three areas - inline, sidebar on break. Inline is in the body copy column, sidebar is obviously in the sidebar and break breaks from the row and gives you access to make something the full screen width. You can technically add more options by making divs in the columns a different percentage width (width: 50%, 150%, etc.) if you need a work around. Try to avoid doing this if you can, but often when making graphics something doesn't line up quite right and you need a little more freedom. Stay away from absolute values (width: 50px) as they are more buggy when coding a responsive design unless something is small like styling the twitter button to width: 5px, since this doesn't need to change when loaded on a mobile device.

If you want to animate your media, look at animate.css and wow.js to reveal those animations when scrolled over. Look at my story about Yaolin Wang's murder or check out the code to see a working example of both.

Embeded

Some content you're just going to want to embed from another service. I highly encourage this and have listed some programs below of things I think are useful. If you do this though, you need to able to create the embeded content on an offical IDS account, so we don't loose access to it next semester. I'd recommend that you make a version on an account you control and then work with a digi editor to export it to an IDS account. Don't try to reinvent the wheel, chances are if google has developed something that does what you're trying to do, it is going to work better than something you make yourself. Many sevices will have an embed option under their share menu, and will generate an iframe for you to paste over. Just copy that into the figure tag, and add a figcaption. Sometimes the sizing is a little wonky, so you might need to adjust width="100%" and play with the height until it looks good.

Guidelines for IDS Submission

Follow all commented instructions and call ids.css and ids.js last in your html. There is no absolute style rulebook, and we're open to changes as long as you have a compelling justification.

Try to keep the digital desk updated on your story throughout your reporting and not just when you're ready to publish. You should start working with us directly three weeks before publication, I’ve included outline with deadlines for each week included in the next section.

The earlier we know how you're going to tell the story the better it will work online. I'm excited if you want a featured added so just let me know if you have any ideas. We will likely be debugging eight to ten stories at the end of the semester, so we need to already be familiar with your code and its issues before then.

When you’re designing your story, keep all your files in your project folder. Story structures jump around, and photos get swapped out late so it is best to have everything in your folder even though it will be large. Still optimize your images for the web, don’t have duplicates, etc. but if there is a chance it will be needed in the final project keep it in. We will work to get your folder slim before publishing.

We host everything on github, but if you're not comfortable working with it you can also just give us your code via email or a thumb drive. It can't include any server-side programming like Ruby, Python, PHP, etc. but I don't imagine you would ever need to. All project folders must be less than 80MB. There is a lot of extra javascript in the template folder as it shows you every option available. The starter folder is trimmed down so start working from there.

File Optimization

Schedule and Deadlines

Start working with our desk roughly three weeks before you want to publish. If you meet a deadline early you can move on to the next week.

Week 1

  • Submit your article at this pitchlist as other: buildout and include:
    • A working headline and slug. Both can change if needed, but the slug will show up in the URL so make sure it isn’t insensitive.
    • A general list of the media you have (e.g. 2 maps, one chart, an audio slideshow and a couple photos).
    • Contact info of everyone working on the story.
    • When you want it to publish. If you’re unsure about the specific date, just put the Monday of the week you want to publish.
  • Make sure you’re signed up for our slack account so we can be in communication throughout the editing process.
  • Come into an office hour (M/W 3-4 p.m.) or schedule a time to meet with me to discuss your story. Think of ideas you want to include in your story that you couldn’t figure out or any big challenges you’re nervous about. We will also go through how to get started with the folder.

Week 2

  • Use the starter folder and copy-paste over examples from the template folder. If you run into issues getting things to work properly send us a message on slack. We might be able to identify the problem or send you a code snippet to fix it.
  • Come into office hours or schedule a time to meet me to discuss more complicated problems.
  • By the end of the week submit a first draft of your article. If you need / want to use lorem ipsum text that’s fine but have your media layed out in the correct hierarchy.

Week 3

We need a full week to edit your code, so make sure to submit a rough draft a five days before your publication date. Every day has specific deadlines this week. You can come into the IDS to work if you want hands on help, or can submit folders remotely.

  • Day 5
    • Tell us about major bugs or style problems you haven’t been able to figure out.
    • Come in to receive an edited rough draft. You will receive two to do lists, one for bugs and other to make it comply with the ids style guide.
    • Submit a final slug for use in the url of your website.
    • Consider reaching out to experts or subjects in your story for promotion on social media. Tell them you plan to publish in a week and ask would they retweet or share your story on facebook. Try sharing your story in multiple niche circles where it will gain the best traction online.
  • Day 4
    • Update your style and submit a new version that complies with the ids style guide.
    • Start debugging
  • Day 3
    • More debugging
    • Remove any features that still aren’t working by the end of the day.
  • Day 2
    • Submit a debugged version. It should work on mobile, medium and large screens. If graphics or features aren’t working, they should be removed and a fully functional version submitted. If you are able to get them working before publication you can go back in and add it in, but we want a stable version that we know we can publish this day.
    • Push your folder to our server so it is live.
    • Make a list of problems discovered once the article went live. Normally these are small like a .js file being called in wrong, or an image being locally hosted somewhere outside the project folder.
  • Day 1 (night of production):
    • Place in copy edited copy and captions.
    • Fix any issues found when the article went live.
    • Possibly add in fixed features that were removed the day before.
    • Prepare article for social media.
  • Day 0 (day is appears in the paper)
    • Sleep
    • Try to be available to contact if other problems pop up or the story gains a lot of traction online.
